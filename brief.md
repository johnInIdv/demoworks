# Client Brief

**Client:** Me — a solutions engineer in B2B software sales.

## What's broken

I build a custom product demo for nearly every prospect, and I build each one by hand from the discovery-call transcript. A good one costs me three to five hours: re-reading the call, deciding which capabilities fit, inventing sample data, and assembling something to show. Under deadline I fall back on a generic template deck — the same screens, the same fake "Acme Corp" data, every time. Those demos land flat, because the prospect doesn't see *their* workflow; they see a brochure. The work also doesn't compound: every transcript starts from zero, and the judgment I apply — which pain maps to which capability, what to admit the product can't do yet — lives only in my head.

## What I've tried

A template library (still generic), copying my best past demos (wrong domain every time), and longer discovery calls (more notes, same manual assembly).

## What I need

A folder I can drop a transcript into that does what I do, in order: triage whether it's worth a deep build, extract the customer's actual workflow, map each pain point to the right capability *with honest limits*, plan the demo, generate realistic sample assets, and produce a sharable one-page preview I can send before the meeting. It should encode the mapping judgment as reusable reference, so the system gets sharper every time I correct it — not start over.
